# X自動投稿システム 運用手順書

プログラミングの知識がなくても運用できるように書いています。

## このシステムは何をしているか

毎日 **12:00 と 21:00（日本時間）** に、GitHub Actions（GitHubに付いている自動実行の仕組み）が起動し、次の順で動きます。

1. **テスト実行** — プログラム自体が壊れていないかを自動チェック。失敗したら投稿せず安全に停止します
2. **投稿選択** — Googleスプレッドシートから「投稿日付が今日」「時間が現在±30分」「ステータスが『済み』でない」行を探します
3. **本文チェック** — 空の本文や長すぎる本文（日本語約140字超）は投稿せずエラーとして記録します
4. **投稿** — Xに投稿します。ネットワーク障害やX側の回数制限のときは、時間を空けて最大3回まで自動で再試行します
5. **「済み」マーク** — 投稿に成功した行のステータス列を「済み」に更新します
6. **結果の記録** — 実行結果を `logs/post_summary.md` に書き込み、GitHubに自動保存します

セルフリプライ列に文章があれば、本編投稿へのリプライとして続けて投稿します。

## 毎日の確認方法（30秒）

GitHubのリポジトリで **`logs/post_summary.md`** を開くだけです。最新の実行が一番上に載っています。

| 表示 | 意味 | やること |
|---|---|---|
| ✅ 投稿成功 | 正常に投稿された | 何もしなくてOK |
| ⏭ この時間帯の投稿なし | シートに該当行がなかった | 投稿予定があったはずなら、シートの日付・時間・ステータスを確認 |
| 🧪 ドライラン | テスト実行（投稿していない） | 何もしなくてOK |
| ⚠️ 一部失敗 | 本編は投稿されたが、リプライか「済み」更新に失敗 | 記録内の「注意」の指示に従う |
| ❌ 失敗 | 投稿されなかった | 記録内の「原因」「対処」を読んで対応 |

さらに詳しいログは GitHub → **Actions** タブ → 「Post Tweet」→ 失敗した実行をクリックすると見られます。

## よくあるトラブルと対処

結果ファイルの「原因」「対処」に具体的な指示が出ますが、代表例：

- **X APIの認証・権限エラー** → GitHubの Settings → Secrets and variables → Actions で、`X_API_KEY` など4つのキーを設定し直す
- **レート制限（回数制限）** → 時間を置けば回復。手動で再実行するか次回を待つ
- **ネットワークエラー** → Actionsタブから該当の実行を開き「Re-run all jobs」で再実行
- **本文が長すぎる／空** → スプレッドシートの該当行の本文を直す。その行は未投稿のまま残っていますが、時間帯が過ぎると選ばれなくなるので、投稿し直したい場合は「投稿日付」「時間」を未来のスロットに設定し直す
- **⚠️「済み」更新に失敗** → 投稿自体は成功しています。シートの該当行のステータスを手動で「済み」にする（二重投稿防止）

## 手動でテスト実行する（ドライラン）

実際には投稿せずに、動作だけ確かめる方法：

1. GitHub → **Actions** タブ → 左の「Post Tweet」
2. 右上の **Run workflow** をクリック
3. 「ドライラン（実際には投稿しない）」に **チェックを入れて** 実行

チェックを入れずに実行すると、その時間帯に該当する投稿があれば**実際に投稿される**ので注意してください。

## システムを止めたいとき

GitHub → Actions タブ → 「Post Tweet」→ 右上の「…」メニュー → **Disable workflow**。
再開は同じ場所の **Enable workflow**。

## 知っておくべき注意点

- **GitHubの仕様で、リポジトリに60日間まったく更新がないと自動実行が止まります。** このシステムは毎回結果ファイルをコミット（保存）するので通常は問題になりませんが、長期間 ⏭（投稿なし）が続くと止まる可能性があります。止まったらActionsタブに「Enable」ボタンが出るので押してください
- GitHub Actionsのcron（定時実行）は数分〜数十分遅れることがあります。±30分の余裕を持たせてあるので、投稿時刻が多少ズレるのは正常です
- スプレッドシートの列名（投稿日付・時間・柱・構文・本文・セルフリプライ・ステータス）は変更しないでください
